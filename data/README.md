# MotoGP Analytics - Data Directory Structure

## 📊 Overview

This directory contains all datasets used in the MotoGP Analytics project, organized according to data science best practices and the CRISP-DM methodology pipeline.

## 🗂️ Directory Structure

```
data/
├── raw/          # Original datasets from Kaggle (immutable)
├── interim/      # Partially processed datasets (Phase 02 outputs)
├── processed/    # Final clean datasets (ready for analysis)
├── external/     # External data sources (future)
└── README.md     # This documentation
```

## 📋 Data Pipeline Flow

### Phase 01: Data Understanding → **raw/**
- **Source**: [Kaggle MotoGP Dataset](https://www.kaggle.com/datasets/alrizacelk/moto-gp-world-championship19492022)
- **Purpose**: Original, unmodified data for exploration and understanding
- **Files**:
  - `constructure_world_championship.csv` - Constructor championship records
  - `grand_prix_events_held.csv` - Event hosting information
  - `grand_prix_race_winners.csv` - Historical race winners
  - `riders_finishing_positions.csv` - Detailed race results
  - `riders_info.csv` - Rider biographical and career data
  - `same_nation_podium_lockouts.csv` - National podium domination

**🚫 Important**: Files in `raw/` should NEVER be modified. They represent the single source of truth.

### Phase 02: Data Preparation → **interim/**
- **Source**: Processed from `raw/` by Phase 02 notebooks
- **Purpose**: Cleaned, standardized, and feature-engineered datasets
- **Transformations Applied**:
  - Column name standardization
  - Data type corrections
  - Missing value treatment
  - Derived metric creation
  - Data validation and consistency checks

**Expected Files** (generated by Phase 02 notebooks):
- `riders_info_prepared.csv` - Clean rider statistics with derived metrics
- `constructors_prepared.csv` - Standardized constructor data
- `race_winners_prepared.csv` - Clean race winner records
- `finishing_positions_prepared.csv` - Standardized race results
- `events_held_prepared.csv` - Clean event hosting data
- `podium_lockouts_prepared.csv` - Processed podium domination data

### Phase 03+: Modeling & Analysis → **processed/**
- **Source**: Further processed from `interim/` for specific analysis needs
- **Purpose**: Analysis-ready datasets optimized for modeling and insights
- **Use Cases**:
  - Machine learning model training
  - Statistical analysis
  - Business intelligence queries
  - Dashboard data feeds

### External Sources → **external/**
- **Purpose**: Data from external APIs, supplements, or third-party sources
- **Future Integration**: Circuit details, weather data, technical specifications
- **Format**: Same preparation pipeline applied (raw → interim → processed)

## 📊 Data Quality Standards

### Raw Data
- **Immutable**: Original files never modified
- **Documented**: Source and acquisition date recorded
- **Validated**: Basic quality checks in Phase 01 notebooks

### Interim Data
- **Standardized**: Consistent column naming and typing
- **Cleaned**: Missing values handled appropriately
- **Enriched**: Derived metrics and features added
- **Validated**: Data consistency and logical relationships checked

### Processed Data
- **Optimized**: Analysis-ready format
- **Documented**: Transformation steps recorded
- **Tested**: Quality assurance validated
- **Versioned**: Reproducible processing pipeline

## 🛠️ Usage Guidelines

### For Notebook Development
```python
from pathlib import Path

# Always use relative paths from notebook location
raw_data_path = Path("../data/raw/")
interim_data_path = Path("../data/interim/") 
processed_data_path = Path("../data/processed/")

# Example: Loading prepared data for modeling
df = pd.read_csv(interim_data_path / "riders_info_prepared.csv")
```

### For Database Integration
- **ETL Scripts**: Use `processed/` data for database import
- **API Endpoints**: Query database, not CSV files directly
- **Cache Strategy**: Database serves as single source of truth

### For Visualization
- **Streamlit App**: Connects to API/database for real-time data
- **BI Tools**: Connect to database for live dashboards
- **Static Reports**: May use `processed/` data for specific analyses

## 🔒 Data Governance

### Version Control
- **Raw data**: Tracked in Git (small files only)
- **Interim/Processed**: Listed in `.gitignore` (generated files)
- **Documentation**: All metadata and lineage tracked

### Quality Assurance
- **Automated Validation**: Each processing step includes quality checks
- **Manual Review**: Data quality assessed in evaluation notebooks
- **Reproducibility**: Complete pipeline documented and executable

### Privacy & Security
- **No PII**: Dataset contains only public racing statistics
- **Attribution**: Kaggle dataset properly credited
- **Licensing**: Academic/research use compliant
